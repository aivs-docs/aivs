基础协议

Xiao AI Voice Service 云端API使用HTTP2.0作为传输协议, 兼容Alexa Voice Service v20160207(简称AVS)，为了更好理解HTTP2传输协议的建立、管理，请参考：

  • Managing an HTTP/2 Connection
  • Structure an HTTP/2 Request

本文只介绍传输消息格式,消息鉴权及交互模型请参考相关文档

消息格式

服务的请求和响应消息都以Multipart的格式封装。 消息中携带的每一个Entity,如果是文本,则以UTF-8编码并保存为JSON格式的文本;如果是音频,则应当是二进制字节流的格式的附件。

注意 1 即使是纯文本的消息,也必须按照Multipart格式封装。即整体表现为一个Multipart格式的消息,消息中只携带一个Entity,Entity内容为一个JSON文本,而不是直接使用JSON文本格式的消息。 注意 2 本文档只说明消息格式，消息鉴权请参考鉴权文档。

请求消息格式

Http Headers

:method = POST or GET
:scheme = https
:path = /v20160207/{{events or directives}}
content-type = multipart/form-data; boundary={{boundary_term}}

method 客户端发送的请求需要使用POST，获取服务下发指令使用GET。 scheme 为了保证安全，只支持https。 path 客户端的请求需要发送到路径/v20160207/events。服务端的指令在路径/v20160207/directives获取。 content-type 如前所述,content-type字段应当永远是multipart/form-data。boundary遵循http协议即可。

Message Body

JSON Entity

客户端的每个请求都需要携带JSON Entity来描述请求信息。

Entity Headers
Content-Disposition: form-data; name="metadata"
Content-Type: application/json; charset=UTF-8
Entity Body
{
    "event":{
        "header": {
            "namespace": "{{STRING}}",
            "name": "{{STRING}}",
            "messageId": "{{STRING}}",
            "dialogRequestId": "{{STRING}}"
        },
        "payload": {
        }
    }
}

客户端的请求放在event里。每个event都必须携带header和payload部分。

header描述event的基本信息。各字段意义如下：

namespace

指明调用接口。

name

请求类型。

messageId

消息Id，唯一标识一条请求。

dialogRequestId

对话Id，多轮对话时用来区别一轮对话。

payload部分携带具体内容，根据接口和请求类型不同携带不同字段，详细参考API接口文档说明。

Binary Entity

客户端请求如需携带音频，在Binary Entity中携带。

Entity Headers
Content-Disposition: form-data; name="audio"
Content-Type: application/octet-stream
Entity Body

音频格式要求见API文档说明。

响应消息格式

Http Headers

content-type = {multipart/form-data; boundary={{boundary_term}} or {application/json; charset=UTF-8}

content-type 如果服务端处理正常，则返回消息为multipart格式，如果处理出现异常，则直接返回JSON格式的文本信息。

Message Body

JSON Entity

服务端的每个响应都会携带JSON Entity来描述响应信息。

Entity Headers
Content-Disposition: form-data; name="json-message"
Content-Type: application/json; charset=UTF-8
Entity Body
{
    "directive":{
        "header": {
            "namespace": "{{STRING}}",
            "name": "{{STRING}}",
            "messageId": "{{STRING}}",
            "dialogRequestId": "{{STRING}}"
        },
        "payload": {
        }
    }
}

服务端的响应放在directive里。与请求相同，每个directive由header部分和payload部分组成。header描述directive的基本信息。payload部分携带具体内容。 具体内容参考API文档说明。

Binary Entity

服务端如果下发了需要客户端直接播放的音频，在Binary Entity中携带。

Entity Headers
Content-Disposition: form-data; name="audio"
Content-Type: application/octet-stream
Content-Id: <{{STRING}}>
Entity Body

音频格式见API文档说明。

Request请求

:method = POST
:scheme = https
:path = /v20160207/events
authorization = Bearer d~APPID~DeviceID~accessToken
content-type = multipart/form-data; boundary=this-is-a-boundary

--this-is-a-boundary
Content-Disposition: form-data; name="metadata"
Content-Type: application/json; charset=UTF-8

{
    "event": {
        "header": {
